Section 01

JOINs

A JOIN combines columns from multiple tables using a common unique identifier, or “key.”

A note on relational databases
SQL's superpower is letting businesses store enormous amounts of data efficiently
That efficiency comes from storing each entity separately, in its own table — no repetition
To get a full picture of the data, we need to JOIN those tables back together
Why Excel doesn't scale
Excel spreadsheets lead to redundancy. Relational databases eliminate it — storing each entity once and linking tables by keys.
The Excel way
One flat table, repeated data

primary key foreign key
Each entity lives in one table. Primary keys identify rows; foreign keys link tables — cheaper, cleaner, scales to billions of rows.

Table: business.tuition_fees · 95 rows
business.business_schools ⨝ business.tuition_fees
SELECT   bs.*
       , tf.*
FROM     business.business_schools AS bs
JOIN     business.tuition_fees     AS tf
  ON     bs.school_rank = tf.school_id;

How do the tuition fees compare between top MBA schools?

1
AS bs and AS tf alias the long table names so we can refer to them concisely
2
JOIN stitches the two tables together row-by-row
3
ON bs.school_rank = tf.school_id is the matching key — rows join where these values are equal
Heads up
Only 95 rows returned — not every school in business_schools has a row in tuition_fees. INNER JOIN silently drops the unmatched ones.
business.business_schools ⨝ business.tuition_fees
SELECT   bs.*
       , tf.*
FROM     business.business_schools AS bs
LEFT JOIN business.tuition_fees     AS tf
  ON     bs.school_rank = tf.school_id;

What if we want to keep every school, even those with no tuition data?

1
LEFT JOIN keeps every row from the left table (bs), matched or not
2
Result is at least as big as the left table — could be bigger if there are multiple matches on the right
3
Unmatched rows get NULL values in the right-table columns
Result
100 rows — every school from business_schools is retained; the 5 without tuition data show NULL
Essential JOIN-ing tips
01
Know your keys
Understand the primary keys of each table and how they link together — that's what you'll JOIN ON
02
Sense-check the size
Always check the row count of the result — e.g. an INNER JOIN should never produce more rows than the smaller of the two input tables
03
Alias sensibly
Use short, meaningful aliases — typically an abbreviation of the original table name (e.g. business_schools → bs)

Solutions
Q1 · Film + language
SELECT f.title
     , f.rating
     , f.rental_rate
     , l.name AS language
FROM   dvdrental.film AS f
INNER JOIN dvdrental.language AS l
  ON f.language_id = l.language_id
ORDER BY f.title;
Q2 · Rentals per customer
SELECT c.first_name
     , c.last_name
     , COUNT(r.rental_id) AS rental_count
FROM   dvdrental.customer AS c
LEFT JOIN dvdrental.rental AS r
  ON c.customer_id = r.customer_id
GROUP BY c.customer_id
       , c.first_name
       , c.last_name
ORDER BY rental_count DESC;
Q3 · Actors in PG films
SELECT f.title
     , a.first_name
     , a.last_name
FROM   dvdrental.film AS f
JOIN   dvdrental.film_actor AS fa
  ON f.film_id   = fa.film_id
JOIN   dvdrental.actor AS a
  ON fa.actor_id = a.actor_id
WHERE  f.rating = 'PG'
ORDER BY f.title
       , a.last_name;
Q4 · Customer + city
SELECT c.first_name
     , c.last_name
     , ci.city
FROM   dvdrental.customer AS c
JOIN   dvdrental.address AS a
  ON c.address_id = a.address_id
JOIN   dvdrental.city AS ci
  ON a.city_id    = ci.city_id
ORDER BY ci.city
       , c.last_name;
Q5 · Films per category
SELECT cat.name AS category
     , COUNT(f.film_id) AS film_count
FROM   dvdrental.film AS f
JOIN   dvdrental.film_category AS fc
  ON f.film_id      = fc.film_id
JOIN   dvdrental.category AS cat
  ON fc.category_id = cat.category_id
GROUP BY cat.name
ORDER BY film_count DESC;

Section 02

Subqueries

A query inside another query.

Subqueries
A subquery is a query nested inside another query — a parent wraps around it.
The subquery runs first — it returns a value, a row, or a table
The parent query then uses that result — typically in a WHERE, SELECT, or FROM clause
Handy for comparing rows against a summary — e.g. “above the overall average”
business.business_schools
-- Find schools with salary above the overall average.
-- Show: school_name, weighted_salary_usd (highest first).

SELECT   school_name
       , weighted_salary_usd
FROM     business.business_schools
WHERE    weighted_salary_usd > (
           SELECT AVG(weighted_salary_usd)
           FROM   business.business_schools
         )
ORDER BY weighted_salary_usd DESC;

Which schools earn above the overall average?

1
The subquery (in parentheses) runs first — returns a single number: the overall AVG salary
2
The parent query then filters WHERE weighted_salary_usd > [that number]
3
Much cleaner than running two queries and copy-pasting the result — and it always reflects the latest data

A subquery is just a regular SELECT statement nested inside another query, wrapped in parentheses.

A subquery in the FROM clause behaves like a temporary table — and needs an alias so the outer query can refer to it.

The outer query is evaluated first, then the subquery runs to provide additional filters or columns.

Which of these best describes a window function?

A
It collapses rows into a single result per group — like GROUP BY, just with different syntax.
B
It performs a calculation across a set of rows related to the current row, without collapsing the result.
C
A subquery that runs inside the WHERE clause to filter rows.
D
A way to join two tables together using a shared key.
business.business_schools
-- Rank schools by salary, within each country.

SELECT   school_name
       , primary_location AS country
       , weighted_salary_usd
       , RANK() OVER (
           PARTITION BY primary_location
           ORDER BY     weighted_salary_usd DESC
         ) AS country_rank
FROM     business.business_schools
ORDER BY country
       , country_rank;

Which school ranks #1 in each country by salary?

1
RANK() is the window function — it assigns a rank to each row
2
PARTITION BY primary_location resets the rank for each country — each country starts at 1
3
ORDER BY weighted_salary_usd DESC defines what “highest” means within each partition
4
Every row from the source table is kept — you just gain a new country_rank column
